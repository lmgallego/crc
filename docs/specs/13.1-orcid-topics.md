# 13.1 — Auto-clasificación de topics y mejora UX de filtros

Sub-spec del 13. Resuelve dos problemas detectados tras ejecutar el spec 13:

1. **276 papers sincronizados desde ORCID, 0 con topics asignados** → filtros por tema "no funcionan" desde la perspectiva del usuario.
2. **Falta de feedback visual** cuando un filtro no produce resultados → confusión.

## Problema 1: topics vacíos

El spec 13 dejó por diseño que `topics` viniera del overlay manual (`publications-overlay.json`), porque ORCID no devuelve esa información. El overlay está vacío. Resultado: ningún paper queda clasificado.

**Solución: heurística de inferencia automática por keywords en título + journal**, con el overlay manual ganando como override.

### Heurística verificada

Probada sobre 25 títulos reales del Word del CRC:
- **80% match exacto** con la clasificación humana esperada
- **96% cobertura útil** (al menos un topic correcto)
- Fallback a `'training'` (categoría más amplia) cuando ningún keyword matchea

```typescript
function inferTopics(title: string, journal: string): PublicationTopic[] {
  const text = `${title} ${journal}`.toLowerCase();
  const topics: PublicationTopic[] = [];

  for (const [topic, keywords] of Object.entries(TOPIC_KEYWORDS) as [PublicationTopic, string[]][]) {
    if (keywords.some(kw => text.includes(kw))) {
      topics.push(topic);
    }
  }

  return topics.length > 0 ? topics : ['training'];
}
```

## Problema 2: UX de filtros sin feedback

### Cambios en el componente de filtros de `/publicaciones`

Buscar el componente que renderiza los chips de tema (probablemente `components/blocks/publications/publication-filters.tsx` o similar).

#### 2.1 Contadores junto a cada chip

Cada chip muestra el número de papers que tendría ese filtro **considerando los demás filtros activos**:

```
FISIOLOGÍA (47)   BIOMECÁNICA (12)   NUTRICIÓN (18)   ...
```

Implementación:

```typescript
// En el componente padre, calcular conteos por topic considerando otros filtros activos
const topicCounts = useMemo(() => {
  const counts: Record<PublicationTopic, number> = { /* ... */ };
  
  const publicationsAfterOtherFilters = publications.filter(p => {
    // aplica TODOS los filtros activos EXCEPTO topic
    return matchesAuthor(p, activeAuthors) && matchesYear(p, activeYears);
  });
  
  for (const pub of publicationsAfterOtherFilters) {
    for (const topic of pub.topics) {
      counts[topic] = (counts[topic] ?? 0) + 1;
    }
  }
  return counts;
}, [publications, activeAuthors, activeYears]);
```

#### 2.2 Chips vacíos atenuados

Si `topicCounts[topic] === 0`:
- `opacity: 0.4`
- `pointer-events: none` (no se puede clickar)
- Cursor `not-allowed`
- Aria-disabled

Esto previene la situación: usuario filtra por autor X + topic Y donde no hay intersección.

#### 2.3 Mensaje cuando hay 0 resultados

Si tras aplicar filtros `filteredPublications.length === 0`, en lugar de mostrar la lista vacía:

```
┌────────────────────────────────────────────────────────────┐
│                                                            │
│  No hay publicaciones con estos filtros.                   │
│                                                            │
│  Probá quitar algún filtro.                                │
│                                                            │
│  [Limpiar filtros]                                         │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

Tipografía: la misma que el resto de placeholders del sitio. **NO inventar componentes nuevos**, reutilizar lo que ya existe (probablemente un `EmptyState` o similar). Si no existe, crearlo con el mismo lenguaje visual de los demás vacíos del proyecto.

### Mismo patrón para chips de autor

Aplicar la misma lógica (contador + atenuación) a los chips de autor CRC. Si Mikel + Biomecánica no tiene intersección, el chip de "Mikel Zabala" se atenúa al activar Biomecánica.

## Reversibilidad

**Mismo patrón que el spec 13**: rama dedicada `feat/spec-13-1-topics`. La inferencia automática es una función pura nueva; si no convence, se aísla detrás de un flag adicional:

```typescript
// lib/config/features.ts
export const FEATURES = {
  orcidPublications: true,
  orcidAutoTopics: true,  // ← NUEVO. Si false, no infiere, topics quedan []
};
```

Y en el script:

```typescript
import { FEATURES } from '../lib/config/features';

const topics = FEATURES.orcidAutoTopics 
  ? inferTopics(title, journal)
  : [];
```

## Limitaciones conocidas

1. **La heurística no es perfecta**. ~20% de los papers tendrán al menos un topic discutible. Aceptable porque:
   - Cubre el caso de uso principal (filtros funcionando con cifras razonables)
   - El overlay manual permite corregir caso por caso
   - Los 2 seeds clave (Athlete 2.0, Cycling 3.0) ya tienen curación manual

2. **Algunos papers caerán en `training` por defecto** cuando no haya keyword reconocido. Es el mejor fallback porque `training` es la categoría más amplia y mejor representada en el CRC.

3. **No infiere `ai`** salvo en papers muy explícitos. Ningún paper del listado actual matchea `ai` (excepto Cycling 3.0 por el overlay manual). Esto está bien — refleja la realidad: la línea AI es nueva en el CRC.
